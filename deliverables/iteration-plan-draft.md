# EpochJudge 战略迭代计划（草案）

## 产品定位
EpochJudge 是一套可自托管的开源在线评测系统，适用于信息学竞赛（OI）、ACM 练习、校内机考与训练营等场景。

## 技术架构现状
- **前端**：React + Rspack + Ant Design
- **后端**：NestJS
- **数据库**：MySQL + Prisma
- **缓存/队列**：Redis + BullMQ
- **架构**：Monorepo (Yarn 4 workspaces)

## 当前功能模块
1. **题库系统**：题目浏览、Markdown 题面、限时/限内存、公开/隐藏可见性
2. **提交系统**：多语言代码提交（JS/Python/Java/C/C++）、提交记录列表、评测进度
3. **评测系统**：OI/ACM 模式、队列调度、分布式判题
4. **比赛系统**：比赛列表、题单、积分榜（OI/ACM）
5. **管理系统**：题目 ZIP 导入、用户 CSV 导入、判题节点状态查看
6. **用户系统**：注册登录、个人设置、数据面板

## 风险评估与应对策略

### 技术风险
1. **性能瓶颈风险**：
   - **风险**：随着用户增长，系统性能可能下降
   - **应对**：提前进行性能测试，优化数据库查询，增加缓存策略

2. **安全风险**：
   - **风险**：代码执行环境可能存在安全漏洞
   - **应对**：加强沙箱隔离，定期安全审计，及时更新依赖

3. **兼容性风险**：
   - **风险**：新功能可能与现有系统不兼容
   - **应对**：充分测试，渐进式发布，保持向后兼容

### 市场风险
1. **竞争风险**：
   - **风险**：现有OJ系统功能完善，用户迁移成本高
   - **应对**：差异化定位，提供独特价值，降低迁移成本

2. **用户接受度风险**：
   - **风险**：用户可能不愿意尝试新系统
   - **应对**：提供试用版本，收集用户反馈，持续改进

3. **商业模式风险**：
   - **风险**：开源项目盈利模式不清晰
   - **应对**：探索多种盈利模式，平衡开源与商业

### 运营风险
1. **社区运营风险**：
   - **风险**：开源社区活跃度不足
   - **应对**：积极运营社区，激励贡献者，提供良好文档

2. **维护成本风险**：
   - **风险**：项目维护成本高
   - **应对**：自动化测试，持续集成，代码质量保证

3. **人才流失风险**：
   - **风险**：核心开发者可能离开
   - **应对**：培养团队，知识共享，文档完善

## 总结

EpochJudge作为开源OJ系统，具有自托管、数据可控、技术栈现代等优势。通过分阶段迭代，先完善核心功能和用户体验，再开发差异化功能和建设社区，最后探索生态扩展和商业化，可以逐步建立竞争优势，吸引更多用户和贡献者，形成良性发展循环。

1. **用户体验**：提供流畅、直观的用户体验

3. **社区活跃**：建设活跃的开源社区
4. **差异化**：提供独特的价值主张
5. **可持续**：建立可持续的商业模式
