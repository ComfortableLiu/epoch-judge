## Context

EpochJudge 前端使用 React + React Router 构建单页应用。当前路由配置中，未匹配的路由会被重定向到首页，用户无法感知页面不存在。数据加载时使用全局 spinner，导致页面布局跳动，用户体验不佳。

当前状态：
- 路由配置位于前端路由文件中
- 列表页（题目、比赛、提交）使用简单的 loading 状态
- 无专门的 404 页面组件

## Goals / Non-Goals

**Goals:**
- 提供友好的 404 页面，明确告知用户页面不存在
- 为用户提供返回首页的便捷入口
- 为主要列表页添加骨架屏 Loading，提升加载感知
- 保持与现有 Ant Design 设计风格一致

**Non-Goals:**
- 不修改后端 API 或数据库结构
- 不实现复杂的错误边界或错误上报
- 不为所有页面添加骨架屏（仅限主要列表页）

## Decisions

### 1. 404 页面实现方案

**选择：React Router 通配路由 + 独立 NotFound 组件**

在路由配置中添加 `path: "*"` 通配路由，匹配所有未定义路径。创建独立的 `NotFound` 组件，包含友好提示和返回首页链接。

**备选方案：**
- 重定向到首页并显示 toast 提示：用户体验不够直观，无法明确告知页面不存在
- 使用 Error Boundary：过于重量级，404 不是运行时错误

**理由：**
- 符合 React Router 最佳实践
- 组件独立，易于维护和测试
- 用户体验清晰直观

### 2. 骨架屏实现方案

**选择：Ant Design Skeleton 组件**

使用 Ant Design 内置的 `Skeleton` 组件，为每个列表页创建对应的骨架屏组件。

**备选方案：**
- 自定义 CSS 骨架屏：开发成本高，维护困难
- 第三方骨架屏库：增加依赖，与 Ant Design 风格可能不一致

**理由：**
- Ant Design 已包含 Skeleton 组件，无需额外依赖
- 与现有 UI 风格保持一致
- 支持自定义形状和动画

### 3. 骨架屏集成方式

**选择：条件渲染 + 状态驱动**

在列表页组件中，根据 loading 状态条件渲染骨架屏或实际内容。使用 `loading` 状态变量控制显示。

**备选方案：**
- HOC 包装：增加复杂度，不够灵活
- React.Suspense：需要配合 lazy loading，改动较大

**理由：**
- 实现简单直观
- 与现有代码结构兼容
- 易于理解和维护

## Risks / Trade-offs

**[风险] 骨架屏与实际内容布局不一致**
→ 缓解：仔细分析每个列表页的实际布局，确保骨架屏结构匹配

**[风险] 404 页面可能被恶意利用**
→ 缓解：404 页面仅显示静态内容，不暴露系统信息

**[权衡] 骨架屏增加前端代码量**
→ 接受：提升用户体验的必要代价，代码量增加可控

**[权衡] 仅为主要列表页添加骨架屏**
→ 接受：优先覆盖高频访问页面，后续可按需扩展
