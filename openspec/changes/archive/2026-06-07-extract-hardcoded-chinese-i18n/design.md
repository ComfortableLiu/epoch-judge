## Context

前端已实现 i18n 架构，支持 zh-CN 和 en-US 双语切换。使用 react-i18next 库，翻译文件位于 src/locales/ 目录。但部分页面组件中存在硬编码中文文本，导致切换语言后仍有中文显示。

现有问题示例：
- ContestDetailPage.tsx: "官方榜单"、"打星队伍"、"比赛密码"
- 管理后台组件: 各种表单标签、按钮文本、提示信息

## Goals / Non-Goals

**Goals:**
- 全面扫描前端组件，识别所有硬编码中文字符串
- 将硬编码文本提取到翻译文件中
- 在组件中使用 t() 函数替代硬编码文本
- 确保中英文翻译文件完整覆盖所有 UI 文本
- 保持现有 i18n 架构和翻译文件结构不变

**Non-Goals:**
- 不改变 i18n 库或架构
- 不添加新的语言支持（仅 zh-CN 和 en-US）
- 不修改后端 API 消息的国际化（已有独立机制）
- 不重构组件结构或逻辑

## Decisions

### Decision 1: 扫描策略

**选择**: 使用正则表达式和手动检查相结合的方式扫描硬编码中文

**理由**:
- 正则表达式可以快速定位包含中文字符的 JSX 文本
- 手动检查可以识别动态生成的中文文本（如模板字符串）
- 组合方式兼顾效率和准确性

**替代方案**:
- 方案 A: 纯手动检查 → 准确但耗时
- 方案 B: 纯自动化工具 → 可能遗漏动态文本
- 方案 C: 组合方式 → 平衡效率和准确性 ✓

### Decision 2: 翻译键命名规范

**选择**: 使用 `模块.组件.功能.文本类型` 的命名规范

**理由**:
- 层级清晰，便于管理和查找
- 避免命名冲突
- 符合现有翻译文件结构

**示例**:
- `contest.detail.officialRanking` → "官方榜单"
- `contest.detail.starTeams` → "打星队伍"
- `contest.detail.contestPassword` → "比赛密码"

### Decision 3: 提取粒度

**选择**: 按组件为单位提取，同一组件内的相关文本归为一组

**理由**:
- 保持组件内聚性
- 便于维护和查找
- 符合现有翻译文件组织方式

**替代方案**:
- 方案 A: 按页面提取 → 翻译文件过大
- 方案 B: 按功能模块提取 → 可能导致跨组件依赖
- 方案 C: 按组件提取 → 平衡粒度和内聚性 ✓

## Risks / Trade-offs

**[Risk] 遗漏硬编码文本** → 缓解：使用多种扫描方式（正则、手动、代码审查）交叉验证

**[Risk] 翻译键命名冲突** → 缓解：建立命名规范文档，使用命名空间隔离

**[Risk] 影响现有功能** → 缓解：逐步提取，每完成一个组件后运行测试验证

**[Trade-off] 提取粒度** → 选择按组件提取，牺牲少量查找便利性换取更好的内聚性

## Migration Plan

1. 建立翻译键命名规范文档
2. 扫描并列出所有硬编码中文文本
3. 按优先级排序（用户可见页面优先）
4. 逐组件提取并测试
5. 更新翻译文件并验证双语显示
6. 代码审查确认无遗漏
