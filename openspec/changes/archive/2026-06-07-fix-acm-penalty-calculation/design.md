## Context

当前 `scoreboard()` 实现位于 `src/modules/contests/contests.service.ts`，仅查询 `ACCEPTED` 状态的提交来计算罚时。这导致 ACM 模式下罚时计算不准确，因为标准 ACM 规则要求将失败提交也计入罚时。

现有实现流程：
1. 查询所有 ACCEPTED 提交
2. 按用户分组，计算每道题的通过时间
3. 累加通过时间作为总罚时

## Goals / Non-Goals

**Goals:**
- 修正 ACM 模式下罚时计算，符合标准 ACM 规则
- 保持 OI 模式计分逻辑不变
- 确保计分板 API 返回准确的罚时数据

**Non-Goals:**
- 不改变计分板的排序逻辑（仍按罚时升序排列）
- 不改变 OI 模式的计分方式
- 不修改前端显示逻辑（仅后端数据修正）

## Decisions

### Decision 1: 查询所有提交记录而非仅 ACCEPTED

**选择**: 修改查询条件，获取比赛期间所有提交记录（包含所有状态）

**理由**:
- 需要统计失败提交次数来计算罚时
- 仅查询 ACCEPTED 无法获取失败提交信息
- 性能影响可控，因为比赛期间提交量通常有限

**替代方案**:
- 方案 A: 分两次查询（一次 ACCEPTED，一次非 ACCEPTED）→ 增加数据库查询次数，代码复杂度高
- 方案 B: 一次查询所有提交 → 实现简单，性能可接受 ✓

### Decision 2: 在内存中按用户和题目分组统计

**选择**: 将所有提交记录加载到内存，按 `(userId, problemId)` 分组后计算

**理由**:
- 提交记录量通常不大（单场比赛几百到几千条）
- 内存分组比多次数据库查询更高效
- 代码逻辑清晰，易于维护

**替代方案**:
- 方案 A: 使用 SQL GROUP BY 聚合 → 需要复杂 SQL，难以实现"首次 AC 前的失败次数"逻辑
- 方案 B: 内存分组计算 → 实现简单，逻辑清晰 ✓

### Decision 3: 罚时计算公式

**选择**: `penalty = first_ac_time + 20 * failed_count_before_ac`

**理由**:
- 符合标准 ACM-ICPC 规则
- `first_ac_time`: 从比赛开始到首次通过的分钟数
- `failed_count_before_ac`: 首次 AC 前的失败提交次数（仅统计该题）

**替代方案**:
- 方案 A: 包含所有失败提交（包括 AC 后的）→ 不符合 ACM 规则
- 方案 B: 仅统计首次 AC 前的失败提交 → 符合 ACM 规则 ✓

## Risks / Trade-offs

**[Risk] 性能影响** → 缓解：比赛期间提交量有限，内存分组计算性能可接受。如遇大规模比赛，可考虑添加 Redis 缓存。

**[Risk] 数据一致性** → 缓解：使用事务确保提交记录读取的一致性快照。

**[Trade-off] 内存 vs 数据库查询** → 选择内存计算，牺牲少量内存换取更简洁的代码和更少的数据库查询。
