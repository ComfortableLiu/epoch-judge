## Context

Judge Worker 中的 `problemTestcasesCache` 是一个普通的 Map，用于缓存题目的测试用例数据，避免重复从数据库或文件系统读取。随着题库数量增长，该缓存会无限增长，长期运行可能导致内存溢出（OOM）。

现有实现：
- 使用 `Map<string, Testcase[]>` 存储缓存
- 缓存键为题目 ID，值为测试用例数组
- 无淘汰策略，缓存只增不减

## Goals / Non-Goals

**Goals:**
- 实现 LRU 淘汰策略，限制缓存最大条目数
- 防止内存无限增长，避免 OOM
- 保持现有缓存命中逻辑不变
- 最小化性能影响

**Non-Goals:**
- 不改变缓存键值结构
- 不改变测试用例的读取逻辑
- 不添加缓存统计或监控功能
- 不实现分布式缓存

## Decisions

### Decision 1: 实现方式选择

**选择**: 使用 `lru-cache` 库

**理由**:
- `lru-cache` 是成熟的 npm 包，经过广泛测试
- 提供完整的 LRU 实现，包括 TTL、大小限制等功能
- API 简单易用，与现有代码集成方便
- 维护活跃，有良好的社区支持

**替代方案**:
- 方案 A: 使用 Map 插入顺序特性手动实现 → 需要额外代码维护顺序，容易出错
- 方案 B: 使用 WeakMap → 无法控制淘汰时机，不适合固定大小缓存
- 方案 C: 使用 `lru-cache` 库 → 成熟稳定，功能完整 ✓

### Decision 2: 缓存大小设置

**选择**: 默认最大 1000 个条目

**理由**:
- 1000 个题目缓存通常足够覆盖活跃题目
- 内存占用可控（假设每个题目测试用例约 100KB，总计约 100MB）
- 可通过环境变量配置，便于调整

**替代方案**:
- 方案 A: 固定 500 个条目 → 可能不够用，导致频繁缓存未命中
- 方案 B: 固定 2000 个条目 → 内存占用较大
- 方案 C: 默认 1000，可配置 → 平衡内存和性能 ✓

### Decision 3: TTL 策略

**选择**: 不设置 TTL，仅依赖 LRU 淘汰

**理由**:
- 测试用例数据通常不会频繁变化
- LRU 淘汰已经能有效控制内存
- 简化实现，减少配置复杂度

**替代方案**:
- 方案 A: 设置 TTL（如 1 小时） → 可能导致频繁重新加载
- 方案 B: 仅 LRU 淘汰 → 简单有效 ✓

### Decision 4: 配置方式

**选择**: 通过环境变量 `JUDGE_CACHE_MAX_SIZE` 配置

**理由**:
- 与现有配置方式一致（使用环境变量）
- 便于部署时调整，无需修改代码
- 可设置合理默认值

## Risks / Trade-offs

**[Risk] 缓存未命中增加** → 缓解：设置合理的缓存大小（1000），覆盖大多数活跃题目

**[Risk] lru-cache 库依赖** → 缓解：lru-cache 是成熟库，有良好的维护和社区支持

**[Trade-off] 内存 vs 性能** → 选择限制内存使用，接受少量缓存未命中换取内存安全

## Migration Plan

1. 安装 `lru-cache` 依赖
2. 修改缓存实现，将 Map 替换为 LRU 缓存
3. 添加环境变量配置支持
4. 测试缓存功能和淘汰行为
5. 部署并监控内存使用情况
